### Крок шостий - PM2 
Оця хуйня нам треба шоб наш з вами додаток ранився константно і не їбав ні нам, ні юзвєрям голову:
```
sudo npm i pm2 -g

#Якшо консоль матюкнеться на судо, приберіть його, воно і так зі свистом залітає на наш любімий дроплєт
```
Після цього можна і спробувать запуститься:
```
pm2 start app #міняємо app на назву свого корнєвого файлу сєрвєра (в моєму випадку це index) і буде вам щастя
```
Якшо все гуд, а все має бути гут, треба ще перестрахуваться і зробить так шоб PM2 запускався разом із системою нашого любімого дроплєта:
```
pm2 startup ubuntu
```
І бонусом прілагається ще список команд для PM2 які допоможуть вам в майбутньому якшо будете якісь проблеми вирішувати: 
```
pm2 show app
pm2 status
pm2 restart app
pm2 stop app
pm2 logs #я думаю понятно шо воно показує?
pm2 flush #а оце треба шоб почистить всі логи
```
Тепееееер в нас з вами є можливість подивиться на нашу аплікуху якщо впіхуюріть в адресну строку браузєра айпішнік любімого дроплєта з портом на якому раниться апка.

### Крок сьомий - UFW (Uncomplicated Firewall)
Ну тут все просто. Степ бай степ хуячим команди в консолькє любімого дроплєта і радуємся життю: 
```
sudo ufw enable
sudo ufw status
sudo ufw allow ssh 
sudo ufw allow http 
sudo ufw allow https 
```

### Крок восьмий - NGINX та його конфігурація
В цілому процесс його вствновлення та конфігурації простий, але сам по собі він може вас так виєбать і висушить шо жити не захочється. Більш детально я опишу проблеми які виникли з ним у мене в розділі "Додатки".

Отже, все шо від нас треба на старті: 
```
sudo apt install nginx

sudo nano /etc/nginx/sites-available/default
```

Після другої команди нас закине в такий собі конфіг цього йобаного nginx де нам з вами треба буде знайти рядок з директивою `location /` і в середину фігурних дужок додати наступний код: 
```
proxy_pass http://localhost:5000; #порт звісно вказуємо свій
proxy_http_version 1.1;
proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection 'upgrade';
proxy_set_header Host $host;
proxy_cache_bypass $http_upgrade;
```

Також нам треба вказати в тому ж конфігу ваш домєн для апки, якщо нема, то вказуємо пока шо просто айпішнік любімого дроплєта після директиви `server_name`. І в результаті оце все діло має мати наступний вигляд: 
```
server_name yourdomain.com;

    location / {
        proxy_pass http://localhost:5000; #порт звісно вказуємо свій
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }
```

Тицяємо `Ctrl+X`, погоджуємося зберегти зміни `y` та жмемо `Enter` щоб нарешті вийти звідти.

Тепер давайте перевіримо а чи працює ця йобань по-людські, чи нє: 
```
# Перевіряємо працездатність конфігу
sudo nginx -t

# Робимо nginx рестарт
sudo service nginx restart
```

Якшо клята машина вийобується, перевірте чи все ви правильно закопіпастили і чи не пропустили де крапку з комою.

Ну тупер вже можна подивиться на апку в браузері через айпішнік любімого дроплєта без указанія порта.

На цьому етапі почалася єбля штучного мозку чат гпт яка в свою чергу результатів не дала, але наштовхнула на думку, шо можливо цій тупій машині треба пояснить шо я від неї хочу і де їй ті файли треба взять. Пошуршавши ще трошки причєндалами я таки видавив пару крапель інформації з чату де він мені підказав, шо треба в nginx запхать ще директиву `location` з оцим аліасом де я пропишу як цей шлях повністю виглядає. В іотгє народилися наступні рядки коду в конфігє nginx: 

![image](https://github.com/ohiienko-r/digitalocean-droplet-deploy/assets/109099364/de9c4ae5-ecdf-42d4-a3cc-fbfccc188168)

І шо і думаєте хепі енд? Та хуй там. 404 змінилися на 403, шо в свою чергу вже значить шо мені в доступі відказано. Ну тут я вже на досвіді поліз єабть той штучний мозок чату і в момент коли я вже заїбався і майже не дивився шо він там мені випльовує, воно народило нарешті потрібні мені команди: 
```
#перевірка а чи належать цій групі права на доступ до файлів
sudo chown -R www-data:www-data /root/time-tracker/server/public/assets/

#даєм нашим файлікам пермішн
sudo chmod -R 755 /root/time-tracker/server/public/assets/

#даєм доступ до директорій
sudo chmod o+x /root /root/time-tracker /root/time-tracker/server /root/time-tracker/server/public

#ребутим nginx
sudo systemctl restart nginx
```

І о боги, клієнт нарешті запрацював, але єбать єбать, запроси на сєрвєр не уходять.
